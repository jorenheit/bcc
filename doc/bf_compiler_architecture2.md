# Brainfuck Compiler Architecture

This document describes the high-level architecture of the Brainfuck compiler and runtime system.

The compiler generates Brainfuck code that executes a structured program consisting of functions, blocks, and stack frames.

The design introduces several abstraction layers on top of raw Brainfuck:

- **MacroCells** – logical cells consisting of several Brainfuck cells
- **Stack Frames** – structured activation records for functions
- **Blocks** – control flow units inside functions
- **Meta Blocks** – infrastructure for function calls and returns
- **Primitive IR** – intermediate representation used to generate Brainfuck

---

# Core Components

## Program

The `Program` structure represents the entire compiled program.

Responsibilities:

- Holds all functions
- Stores global variables
- Maintains the global block order
- Provides bootstrap and hatstrap sequences

Each block receives a unique **global block index** used by the runtime dispatcher.

---

## Function

A `Function` represents a callable unit.

Components:

- `FrameLayout` describing the stack frame structure
- A set of `Block`s
- An entry block

Each function manages its own locals and block graph.

---

## Block

A block is the smallest executable control-flow unit.

A block:

- belongs to a function
- has a globally unique block ID
- contains a `primitive::Sequence` representing IR instructions

Execution is controlled by the **TargetBlock** field stored in the frame.

---

# Intermediate Representation (IR)

The compiler does not emit Brainfuck directly.

Instead it produces a sequence of **Primitive Nodes** such as:

- MovePointerRelative
- MovePointerDynamic
- CopyData
- MoveData
- ChangeBy
- ZeroCell
- LoopOpen / LoopClose
- Out

Each primitive generates Brainfuck code.

Deferred values (`primitive::Defer::Int`) allow offsets such as frame sizes to be resolved later using the compilation context.

---

# Compiler

The `Compiler` class constructs the IR and manages:

- active function
- active block
- active primitive sequence
- data pointer position

Responsibilities:

- building functions and blocks
- managing stack frames
- generating meta blocks
- emitting primitive IR

---

# DataPointer

The `DataPointer` tracks the logical position of the BF pointer.

It supports:

Static mode:
- pointer known relative to frame origin

Dynamic mode:
- pointer location depends on runtime value

The pointer also tracks the **active MacroCell field**.

---

# Slot

A `Slot` represents a logical variable.

Properties:

- type (local, global, temporary)
- offset inside frame
- size

Important design rule: Slot::size = number of logical elements
It does **not** represent byte width.

Arrays are represented as multiple logical cells.

---

# FrameLayout

Each function has a stack frame layout.

Frame structure:

```
TargetBlock
RunState
ReturnValue
Local variables...
```

`LocalBaseOffset` marks the start of the local area.

---

# MacroCell

A MacroCell represents a logical cell consisting of multiple Brainfuck cells.

Current layout:
```
FrameMarker
EnterFlag
Runtime0
Runtime1
Value
```


Future layout (planned):
```
FrameId
EnterFlag
Runtime0
Runtime1
Value0
Value1
```


MacroCells allow the runtime to attach metadata to each logical cell.

---

# Key Design Principles

### Logical addressing

All pointer movements operate on **logical cells**, not raw BF cells.

Primitives convert logical offsets into raw BF pointer movements using `fieldCount`.

---

### Helper fields

Helper fields (`Runtime0`, `Runtime1`, `EnterFlag`) must always be reset to zero after use.

---

### Pointer invariance

Certain helper operations must leave the pointer unchanged:

- `setNextBlock`
- `syncGlobalToLocal`
- `syncLocalToGlobal`

This allows them to be safely used inside loops.

---

### Frame invariance

`popFrame()` preserves:

- logical cell offset
- active field

This ensures meta-block logic works correctly.

---

# Execution Model Overview

Execution proceeds as:
```
bootstrap
↓
main dispatch loop
↓
blocks execute
↓
meta blocks handle calls/returns
```

Blocks themselves contain no direct call mechanics.  
Function calls are implemented via **meta blocks** generated by the compiler.
